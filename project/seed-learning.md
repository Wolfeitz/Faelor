# Seed Learning

This file captures reusable lessons learned while developing seeded projects.

Do not copy project-specific lore, characters, plots, rules, locations, or private canon into this file. Extract only tools, workflows, pressure tests, and modeling patterns that can help future projects.

## Current Reusable Lessons

- Seeds often imply hidden infrastructure. Extract the systems, institutions, interfaces, routes, permissions, maintenance needs, and failure modes implied by the creator's premise.
- A creator's first idea may be a user-facing interface, not the underlying truth. Preserve the difference between interface, mechanism, law, source, and interpretation.
- Imported abilities, memories, rules, or source facts often need translation across contexts. Track what carries over, what is reinterpreted, what is locked, and what requires new grounding.
- Access is not comprehension. Comprehension is not integration. Integration is not mastery.
- Categories are tools, not cages. Classes, factions, species, roles, genres, and mechanics should be treated as lenses unless the project makes them binding.
- Strong seed work produces stubs before prose: known facts, inferred facts, open points, implied infrastructure, risks, and suggested build depth.
- Sensitive or model-limited content can be represented with placeholders, intent notes, scene boundaries, and later-fill workflows without forcing the active model to generate blocked prose.
- Background agents and local models are most useful for cheap audits: orphan entities, contradiction scans, source drift, visual drift, derivation gaps, and candidate stub extraction.
- Project guidance is process canon: creator preferences, guardrails, tone, placeholder rules, and repeated workflow corrections should persist outside world canon.
- Agents should capture breadth before depth after major brainstorms, then let the creator choose which stubs deserve expansion.
- Do not treat a constrained topic keyword as the whole request. Preserve the usable surrounding work.

## Promotion Rule

Add a lesson here only when it is reusable across projects. If it only matters to one world, story, ruleset, or canon, keep it in that project's canon or feature packet.
