# Interface Abstraction Lens

Use this lens when a project has menus, systems, dashboards, rituals, divine messages, magical senses, command layers, tool panels, social protocols, source UIs, or any other mediated way of perceiving and acting.

This lens is optional. Do not assume every project has an interface.

## Core Distinctions

- Interface: what a user, character, player, reader, or agent sees.
- Mechanism: what actually performs the work.
- Law: what cannot be violated within the relevant scope.
- Convention: what is customary but changeable.
- Interpretation: what someone believes the interface means.
- Permission: what the actor is allowed to do.
- Capability: what the actor can actually do.

## Pattern

```text
actor -> interface -> command/observation -> mechanism -> result -> interpretation
```

## Use Cases

- LitRPG status screens, skills, classes, achievements, quests, and system messages.
- Game UIs, inventory screens, maps, codices, tooltips, and diegetic HUDs.
- Magical, divine, technological, psychic, bureaucratic, or ritual interfaces.
- Authoring tools, wiki dashboards, campaign control panels, and worldbuilding IDEs.
- Social interfaces such as ranks, titles, uniforms, etiquette, law, and institutional procedure.

## Checks

- Is this interface a convenience layer or binding reality?
- What can the interface hide, simplify, mistranslate, or bias?
- Who designed it, maintains it, controls it, or benefits from it?
- What happens when an actor asks for information outside the default interface?
- What is possible but not exposed?
- What is exposed but misunderstood?
- What happens when two scopes use different interfaces for the same underlying mechanism?

## Output Pattern

For each interface, capture:

- purpose
- audience
- visible elements
- hidden mechanism
- permissions
- failure modes
- scope and overrides
- relationship to canon truth

## Provocateur Pressure Tests

- Are we mistaking UI text for objective truth?
- Are we assuming the default display is the only possible display?
- Are we granting capabilities because the interface is convenient rather than because the mechanism supports them?
- Does the interface create story, gameplay, or continuity consequences?
