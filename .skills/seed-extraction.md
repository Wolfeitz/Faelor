# Seed Extraction Skill

Use this workflow after a creator provides a seed, premise, source package, ruleset, campaign idea, visual brief, or imported corpus.

The goal is to extract reusable project structure without inventing canon too early.

## Inputs

Use whatever exists:

- kickoff answer
- feature brief
- source notes
- chat transcript
- visual brief
- ruleset or source canon description
- prior manuscript, campaign notes, wiki, or database summary

## Steps

1. Separate explicit creator statements from agent inferences.
2. Extract candidate things, realms, interfaces, rules, events, relationships, styles, sources, and outputs.
3. For each candidate, create a lightweight stub instead of finished prose.
4. Mark known facts, inferred facts, open points, implied infrastructure, source needs, and build-depth recommendation.
5. Capture process guidance signals: tone, boundaries, placeholder needs, output expectations, pace, and repeated creator corrections.
6. Identify which lenses may help: source import, interface abstraction, cross-context translation, prototype derivation, timeline continuity, nexus topology, visual style, ruleset adapter, or story lens.
7. Run or request a Provocateur pass before promoting stubs to canon.
8. Create the first feature packet only after the working frame is clear enough to avoid locking in the wrong structure.

## Stub Shape

```markdown
## Name

Status: candidate

Kind:

Known:

Inferred:

Open points:

Implied infrastructure:

Possible lenses:

Build depth:

Promotion gate:
```

## Process Guidance Signals

If the seed reveals collaboration needs, capture them outside canon:

- `project/creator.md` for standing preferences
- `project/guardrails.md` for boundaries, model limits, and placeholder strategy
- `project/creative-guidance.md` for tone, audience, output, and story pressure
- `project/lessons.md` for repeated corrections and workflow lessons

## Build Depth

- `mention`: keep as a note only.
- `stub`: identify and track.
- `sketch`: outline structure and dependencies.
- `reference`: build a reusable canon reference.
- `system`: build rules, validators, indexes, or templates around it.

## Rules

- Do not turn inferences into approved canon.
- Do not force categories that the project has not chosen.
- Do not overbuild every extracted candidate.
- Keep the creator's intended output in view.
- Treat weird or incomplete seeds as material to preserve, not errors to smooth away.
